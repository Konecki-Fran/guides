Local DeepSeek-Coder-V2-Lite with Open WebUI on Ubuntu Studio

End-to-end setup: NVIDIA drivers, Ollama, the DeepSeek coder model, Open WebUI in Docker, and an optional VS Code integration.

Target host: a clean Ubuntu Studio 24.04 install. Goal: a local coding LLM usable from a browser and from an editor, with the model served from your own machine.

Ubuntu Studio 24.04 NVIDIA GPU (recommended) Ollama Open WebUI DeepSeek-Coder-V2-Lite Continue.dev (optional)

1.What you are building

Three pieces working together:

  • Ollama - the inference engine. It loads the model into (V)RAM and exposes a local HTTP API on 127.0.0.1:11434.
  • DeepSeek-Coder-V2-Lite - the model. A 15.7B-parameter Mixture-of-Experts model with only ~2.4B active parameters per token, so it punches well above its VRAM footprint.
  • Open WebUI - a browser front-end (ChatGPT-style) that talks to Ollama. Runs on http://localhost:3000.

Flow at runtime: browser -> Open WebUI (port 3000) -> Ollama API (port 11434) -> GPU -> response streams back.

2.Assumptions and naming

"Open Claw" => Open WebUI. There is no product called Open Claw; the standard browser UI for local LLMs with Ollama support is Open WebUI. This guide targets that. If you meant something else, stop here and clarify.
Base vs Instruct. You asked for DeepSeek-Coder-V2-Lite-Base. The Base model is a raw pretrained model: best for code completion and fill-in-the-middle (what editors like Continue.dev consume). It is not chat-tuned, so it will behave poorly as a conversation partner in Open WebUI.

For the browser chat UI you almost certainly want DeepSeek-Coder-V2-Lite-Instruct. This guide installs both: the Base model for editor autocomplete, and the Instruct model for chat in Open WebUI. Pull whichever ones you want; they share weights internally but are distinct Ollama tags.

Everything below assumes sudo access, a working internet connection, and that commands run in a terminal (Ctrl+Alt+T).

3.Hardware sizing and quantization

DeepSeek-Coder-V2-Lite is a 15.7B-parameter MoE model. Pick the quantization that fits your GPU. Quality drops as you go smaller; Q4_K_M is the usual sweet spot.

Ollama tagFile sizeRecommended VRAMNotes
deepseek-coder-v2:16b-lite-base-q2_K6.4 GB8 GBLowest quality, runs on most modern GPUs
deepseek-coder-v2:16b-lite-base-q4_K_M10 GB12 GBRecommended default. Good quality/size balance
deepseek-coder-v2:16b-lite-base-q5_K_M12 GB16 GBNoticeably sharper than Q4 for code
deepseek-coder-v2:16b-lite-base-q6_K14 GB16-20 GBNear-lossless
deepseek-coder-v2:16b-lite-base-q8_017 GB24 GBFits on 3090/4090. High quality.
deepseek-coder-v2:16b-lite-base-fp1631 GB40 GB+A100 40GB / A6000 48GB territory
No GPU? Ollama will run on CPU with system RAM. Target at least 16 GB RAM free for Q4, 24 GB for Q8. Expect 3-10 tokens/sec rather than 40-80 tokens/sec. It will still work.

Replace -base- with -instruct- in any tag above to get the chat variant. Example: deepseek-coder-v2:16b-lite-instruct-q4_K_M.

4.System prep and updates

Start from a clean Ubuntu Studio desktop session. Open a terminal.

sudo apt update
sudo apt upgrade -y
sudo apt install -y curl wget ca-certificates gnupg lsb-release software-properties-common build-essential git
sudo reboot

After reboot, confirm kernel and distro:

uname -r
lsb_release -a

Ubuntu Studio 24.04 ships with kernel 6.8+ which is fine for current NVIDIA drivers.

5.NVIDIA driver install

Skip this section if you have no NVIDIA GPU (AMD users: install ROCm 7 instead; CPU-only users: jump to step 7).

5.1 Check what's detected

lspci | grep -i nvidia
ubuntu-drivers devices

The second command lists recommended drivers. You want the line marked recommended.

5.2 Install the recommended driver

sudo ubuntu-drivers autoinstall
sudo reboot
Secure Boot. If Secure Boot is enabled, the installer will prompt you to set a one-time MOK password. After the reboot you will see a blue MOK Manager screen: select Enroll MOK -> Continue -> Yes, enter the password you chose, then reboot. If you skip this, the NVIDIA kernel module will not load.

5.3 Verify

nvidia-smi

You should see your GPU, driver version, and CUDA runtime version. If you see command not found or NVIDIA-SMI has failed, see Troubleshooting.

You do not need to install the CUDA Toolkit. Ollama ships its own CUDA runtime. Only the NVIDIA driver on the host is required.

6.Docker and NVIDIA Container Toolkit

Open WebUI is easiest to run in Docker. If you skipped GPU, install only Docker and skip the Container Toolkit subsection.

6.1 Install Docker Engine

sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
https://download.docker.com/linux/ubuntu $(. /etc/os-release; echo "$VERSION_CODENAME") stable" \
| sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

6.2 Let your user run Docker without sudo

sudo usermod -aG docker $USER
newgrp docker
docker run --rm hello-world

The last command should print a welcome message. If it does, Docker works.

6.3 Install the NVIDIA Container Toolkit (GPU users)

This is what lets Docker containers see your GPU.

curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
  sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg

curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

sudo apt update
sudo apt install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker

Verify GPU visibility from inside a container:

docker run --rm --gpus all nvidia/cuda:12.4.0-base-ubuntu22.04 nvidia-smi

If nvidia-smi output appears from inside the container, GPU passthrough works.

7.Install Ollama

This runs Ollama directly on the host (not in Docker). That keeps GPU access simple and lets the editor integration talk to it on 127.0.0.1.

curl -fsSL https://ollama.com/install.sh | sh

The installer drops a systemd service. Confirm:

systemctl status ollama --no-pager
ollama --version

If the service is running, move on.

8.Expose Ollama to Docker

By default Ollama binds to 127.0.0.1:11434. The Open WebUI container cannot reach the host's 127.0.0.1, so we do one of two things. Option A is cleaner and is what this guide uses.

Option A (recommended): keep Ollama on 127.0.0.1, let Docker resolve the host

No Ollama config change needed. In the docker run command for Open WebUI we pass --add-host=host.docker.internal:host-gateway and point the UI at http://host.docker.internal:11434. This stays off your LAN and keeps Ollama local-only.

Option B: bind Ollama on all interfaces

Use this only if you also want other machines on your network to hit Ollama. It opens port 11434 to your LAN.

sudo systemctl edit ollama

Paste:

[Service]
Environment="OLLAMA_HOST=0.0.0.0:11434"

Save and exit, then:

sudo systemctl daemon-reload
sudo systemctl restart ollama
curl http://localhost:11434/api/tags
Firewall reminder. If you take Option B on a laptop, run sudo ufw deny 11434 to block it at the firewall unless you actually want external access.

9.Pull the DeepSeek-Coder-V2-Lite model

Choose tags based on section 3. This pulls the Base variant (for completion) and the Instruct variant (for chat). Skip whichever you do not need.

# Base: for editor autocomplete / fill-in-the-middle
ollama pull deepseek-coder-v2:16b-lite-base-q4_K_M

# Instruct: for chatting in Open WebUI
ollama pull deepseek-coder-v2:16b-lite-instruct-q4_K_M

Confirm they downloaded:

ollama list
Storage. Each tag is ~10 GB at Q4_K_M. Ollama stores models in /usr/share/ollama/.ollama/models. If your root partition is small, move the store before pulling by setting OLLAMA_MODELS in the systemd override (sudo systemctl edit ollama).

10.Smoke test from the CLI

Chat test (uses the Instruct variant):

ollama run deepseek-coder-v2:16b-lite-instruct-q4_K_M "Write a Python function that returns the nth Fibonacci number iteratively."

Completion test (uses the Base variant; expect raw continuation, not a conversation):

ollama run deepseek-coder-v2:16b-lite-base-q4_K_M "def quicksort(arr):"

While a generation is running, open a second terminal and run nvidia-smi -l 1. You should see the ollama process using GPU memory. If VRAM usage stays at zero, the GPU is not being used; check Troubleshooting.

Exit the interactive Ollama prompt with /bye.

11.Install Open WebUI

Single command, using Docker:

docker run -d \
  --name open-webui \
  --restart always \
  -p 3000:8080 \
  --add-host=host.docker.internal:host-gateway \
  -e OLLAMA_BASE_URL=http://host.docker.internal:11434 \
  -v open-webui:/app/backend/data \
  ghcr.io/open-webui/open-webui:main

Check it's up:

docker ps
docker logs -f open-webui

When the logs show Application startup complete, open your browser at:

http://localhost:3000
Why not the all-in-one image? The :ollama bundled image runs Ollama inside the same container. That works, but keeping Ollama on the host makes it reachable from VS Code, other tools, and gives you one place to manage models.

12.First run: admin account and model selection

  1. The first account you create becomes the admin. Sign up with any email and password (data stays on your machine).
  2. Top-left, click the model dropdown. You should see both deepseek-coder-v2:16b-lite-instruct-q4_K_M and ...-base-q4_K_M. If not, see the note below.
  3. Select the Instruct tag for chat. Send a test message.

If the dropdown is empty: Settings -> Admin Settings -> Connections -> Ollama API. Confirm the URL is http://host.docker.internal:11434. Click the refresh icon.

Done. You now have a browser-accessible local coding LLM. Bookmark http://localhost:3000.

13.Optional: VS Code via Continue.dev

This plugs the Base model into your editor for tab-autocomplete, and the Instruct model for the chat sidebar.

  1. Install VS Code (.deb package) or the code snap.
  2. In VS Code, install the Continue extension from the marketplace.
  3. Open the Continue sidebar, click the gear icon, choose Open config file. This opens ~/.continue/config.json.

Replace the models and tabAutocompleteModel sections with:

{
  "models": [
    {
      "title": "DeepSeek Coder V2 (chat)",
      "provider": "ollama",
      "model": "deepseek-coder-v2:16b-lite-instruct-q4_K_M",
      "apiBase": "http://127.0.0.1:11434"
    }
  ],
  "tabAutocompleteModel": {
    "title": "DeepSeek Coder V2 (autocomplete)",
    "provider": "ollama",
    "model": "deepseek-coder-v2:16b-lite-base-q4_K_M",
    "apiBase": "http://127.0.0.1:11434"
  },
  "allowAnonymousTelemetry": false
}

Save, restart VS Code. Start typing code - ghost-text completions should appear. Open the Continue sidebar to chat.

14.Operations cheatsheet

Ollama

sudo systemctl status ollama
sudo systemctl restart ollama
journalctl -u ollama -f
ollama list
ollama ps                # running models
ollama rm <tag>          # delete a model
ollama pull <tag>        # update/get

Open WebUI

docker ps
docker logs -f open-webui
docker restart open-webui
docker stop open-webui
# Update to latest:
docker pull ghcr.io/open-webui/open-webui:main
docker rm -f open-webui
# then re-run the docker run command from section 11

Models live in /usr/share/ollama/.ollama/models. Open WebUI data (chats, accounts) lives in the Docker volume open-webui - confirm with docker volume inspect open-webui. Back up that volume if you care about chat history.

15.Troubleshooting

nvidia-smi not found or "NVIDIA-SMI has failed"

Kernel modules aren't loaded. Usually Secure Boot blocked the MOK enrollment. Reboot, enter BIOS, disable Secure Boot, reboot, then:

sudo apt reinstall nvidia-driver-550   # use the version ubuntu-drivers recommended
sudo reboot
Open WebUI dropdown shows no models

The container cannot reach Ollama. From inside the container:

docker exec -it open-webui curl http://host.docker.internal:11434/api/tags

If that fails, either the --add-host flag was missing from docker run or Ollama is bound only to 127.0.0.1 in a way the gateway can't reach. Fix by either (a) recreating the container with the flag, or (b) switching to Option B in section 8 and setting OLLAMA_BASE_URL=http://172.17.0.1:11434 (the default Docker bridge IP).

Generation runs but nvidia-smi shows 0% GPU

Ollama fell back to CPU. Check its logs:

journalctl -u ollama -n 100 --no-pager | grep -iE "cuda|gpu|rocm"

Common causes: driver version too old for the CUDA runtime Ollama ships, or the model is larger than your VRAM so it offloaded to CPU. Try a smaller quantization from section 3.

Port 3000 or 11434 already in use 
sudo ss -tulpn | grep -E "3000|11434"

For 3000, re-run Open WebUI with a different host port, e.g. -p 3001:8080. For 11434, find the conflicting process and stop it - nothing else on the box should be using that port.

Docker: permission denied on /var/run/docker.sock

You didn't pick up the docker group. Run newgrp docker, or log out and back in.

Model download stalls or fails partway

Ollama resumes: just re-run ollama pull <tag>. If it persistently fails at a specific byte count, check free disk (df -h /usr/share/ollama) and your network.

Very slow first token, fast after that

Normal. Ollama lazy-loads the model on first request and keeps it resident for OLLAMA_KEEP_ALIVE (default 5 minutes). Increase via the systemd override:

Environment="OLLAMA_KEEP_ALIVE=24h"

16.Full uninstall

If you want to roll everything back:

# Open WebUI
docker rm -f open-webui
docker volume rm open-webui
docker rmi ghcr.io/open-webui/open-webui:main

# Ollama
sudo systemctl stop ollama
sudo systemctl disable ollama
sudo rm /etc/systemd/system/ollama.service
sudo rm $(which ollama)
sudo rm -r /usr/share/ollama
sudo userdel ollama
sudo groupdel ollama

# Docker (optional, only if you don't want it anymore)
sudo apt purge -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin nvidia-container-toolkit
sudo rm -rf /var/lib/docker /var/lib/containerd

NVIDIA drivers can stay; they're useful beyond this stack.
