Build a Dev Command Runner in Python

A step-by-step tutorial for Python beginners. We will build a small desktop app that runs five common git and shell commands, with a graphical user interface (GUI) made using tkinter (Python's built-in UI library).

1. What you will build

A small desktop window with a dropdown that lets you pick one of five commands, an optional input field for a search pattern, a working directory chooser, and a Run button. The output of the command appears in a scrollable text area below.

+---------------------------------------------------+ | Dev Command Runner | +---------------------------------------------------+ | Command: [ Recent commits (last 20) v ] | | Argument: [ (disabled when not needed) ] | | Working dir: [ /Users/you/code/myrepo ] | | [Browse] | | [Run] [Clear output] | +---------------------------------------------------+ | $ git log -n 20 --oneline | | (cwd: /Users/you/code/myrepo) | | | | 277f83b09 [PY-6315] Add Zerobounce limit | | 6e189dd39 [ST-1039] new polls implementation | | ... | +---------------------------------------------------+

The five commands the tool runs are the most reusable git and shell commands you will reach for again and again when working in any codebase.

2. Tools you need

Python interpreter

You need Python 3.9 or newer. On macOS, this tutorial uses the python.org installer build because it ships with a modern version of Tk (the toolkit tkinter is built on top of). Apple's built-in Python 3.9 also works, but Tk 8.5 there has visual rendering bugs.

To check which Python you have, open Terminal and run:

python3 --version

To check whether tkinter is available with that interpreter, run:

python3 -c "import tkinter; print(tkinter.TkVersion)"

If you see a number like 8.6, you are good to go. If you see a ModuleNotFoundError instead, your Python build does not include tkinter. See Part 6: Troubleshooting.

A text editor

Anything works: VS Code, Sublime, even TextEdit in plain-text mode. We will write a single file.

A git repository

To actually try the tool you will need a git repo to run commands against. Any project on your machine that is under version control is fine.

Part 1: The git and shell commands

Before writing code, let us understand the five commands the tool will expose. These are commands you can also type by hand in Terminal; the tool just gives them a button.

Command 1: git log --oneline master..HEAD

Shows you the commits on your current branch that are not yet on master. In other words: "what work have I done on this branch?" Each commit appears as a single line, formatted as SHORTHASH commit message.

  • git log is git's command for listing commit history.
  • --oneline compresses each commit into one line.
  • master..HEAD means "commits reachable from HEAD (where I am now) but not from master." The two dots are the key; they are how git expresses "the difference between."

Command 2: git diff master...HEAD --stat

Shows you a list of every file that has changed on the current branch versus master, and how many lines were added or removed in each. This is great for a quick "what does this PR touch?" overview.

  • git diff normally shows a line-by-line diff. With --stat, it shows just a per-file summary instead.
  • master...HEAD (three dots, called the "symmetric difference") means "everything that has diverged between master and HEAD." It is the form most often used for branch-vs-main comparisons.
Two dots vs three dots: A..B means commits in B but not A. A...B means commits in either A or B but not both. For git log, you usually want two dots. For git diff against a branch, three dots is the conventional choice.

Command 3: git log --all --oneline | grep -i -E -- 'PATTERN'

Searches every commit message in the entire repo for a keyword or regex. Useful for finding "where did we touch ticket BT-542?" or "did anyone ever commit a fix for that bug?"

  • git log --all includes every branch, not just the current one.
  • | is the shell pipe: it sends the output of the left command into the input of the right command.
  • grep filters lines that match a pattern.
  • -i means case-insensitive.
  • -E enables extended regular expressions, so | inside the pattern means "or" (e.g. BT-542|BT-566).
  • -- is a separator that tells grep "everything after this is a pattern, not a flag." It prevents weird patterns starting with - from being misinterpreted.

Command 4: grep -rnE --exclude-dir=... -- 'PATTERN' .

Recursively searches every file under the current directory for a regex. Like a poor man's "find usages" feature.

  • -r means recursive (descend into subdirectories).
  • -n prints line numbers next to each match (so you can jump straight to the right place).
  • -E enables extended regex, same as before.
  • --exclude-dir=.git and friends skip directories that are huge and full of binary or generated files. Without these excludes, the search reads pack files inside .git and can take minutes.
  • . at the end means "search starting from here." You could replace it with any path.
Why excludes matter: a plain grep -rn pattern . on a real repo can spend most of its time inside .git/objects or node_modules. The output is unreadable and the search may even time out. The --exclude-dir flags are not optional polish; they are what makes the command actually usable.

Command 5: git log -n 20 --oneline

Shows the most recent 20 commits on your current branch. -n 20 caps the count. This is the "what was just done here?" command for orienting yourself in an unfamiliar repo.

Part 2: A focused Python primer

You do not need to know all of Python to follow along, but it helps to recognize a few patterns that show up in our code.

Variables, strings, and f-strings

name = "Fran"
greeting = "Hello, " + name           # concatenation
greeting = f"Hello, {name}"           # f-string (string interpolation)

Lists and dictionaries

A list is an ordered sequence. A dictionary maps keys to values. Both are written with literal syntax:

numbers = [1, 2, 3, 4]
person = {"name": "Fran", "role": "backend dev"}
print(person["name"])   # "Fran"

In our tool, the five commands are stored in a list of dictionaries. Each dictionary describes one command:

COMMANDS = [
    {
        "label": "1. Commits on current branch (vs master)",
        "template": "git log --oneline master..HEAD",
        "arg_label": None,
    },
    # ...
]

Functions and classes

A function is a reusable block of code:

def greet(name):
    return f"Hello, {name}"

print(greet("Fran"))

A class is a blueprint for an object that bundles data and behavior together. We will define a DevRunner class for our app:

class DevRunner:
    def __init__(self):
        self.title = "Hello"

    def show(self):
        print(self.title)

app = DevRunner()
app.show()   # prints "Hello"
  • __init__ is the constructor; it runs automatically when you create an instance.
  • self refers to the instance itself. It is how methods access the object's data.

Importing modules

Python's standard library has thousands of modules. You bring them in with import:

import os                              # bring in the whole os module
from tkinter import ttk                # bring in just one name from a module
import tkinter as tk                   # bring it in with a shorter alias

The if __name__ == "__main__": idiom

The bottom of nearly every Python script ends with this guard:

if __name__ == "__main__":
    DevRunner().mainloop()

This means "only run this code if the file is being executed directly, not if some other Python file imported it." It is a Python convention that lets a file be both a runnable program and an importable library.

Part 3: Meet tkinter

tkinter is Python's built-in module for making desktop GUIs. It is a thin wrapper around an older toolkit called Tk, which itself comes from the Tcl programming language. Tkinter ships with most Python installations, so you usually do not need to install anything extra.

The big picture

Every tkinter program follows the same shape:

  1. Create a root window (the top-level window).
  2. Create widgets (buttons, labels, text fields) and place them inside frames.
  3. Wire up callbacks so the widgets react to clicks and typing.
  4. Call mainloop(). This starts the event loop: tkinter waits for user input and dispatches events.

tk vs ttk

You will see two flavors of widgets:

  • tk.Button, tk.Label ... the original "classic" widgets.
  • ttk.Button, ttk.Label ... "themed" widgets that look native on each operating system.

For a modern look, prefer ttk wherever possible. Some widgets only exist in classic Tk (e.g. tk.Text and the multi-line scrolled text widget we use for output).

Geometry managers: pack vs grid

Tkinter has two main ways to arrange widgets inside a parent:

  • pack: stack widgets along an edge (top, bottom, left, right). Easy for simple layouts.
  • grid: arrange widgets in rows and columns, like a spreadsheet. Better for forms.

You can mix them in the same program, but not inside the same parent container. We will use grid for the controls panel (rows of label+input pairs) and pack for the top-level layout.

The widgets we use

Part 4: Build the tool, piece by piece

Now we put it all together. The full file is about 160 lines. We will introduce it in chunks. By the end of this section you will have the entire program.

Step 4.1: The shebang and imports

#!/Library/Frameworks/Python.framework/Versions/Current/bin/python3
"""Dev Command Runner: tkinter UI to run common git/exploration commands."""

import os

os.environ.setdefault("TK_SILENCE_DEPRECATION", "1")

import shlex
import subprocess
import threading
import tkinter as tk
from tkinter import ttk, scrolledtext, filedialog

Walking through this:

  • The #! line at the very top is a "shebang." If you make the file executable, the shell will use this interpreter to run it. We point at the python.org Python because it has a modern Tk.
  • The triple-quoted string just below the shebang is a module docstring: a description of what this file does.
  • os.environ.setdefault("TK_SILENCE_DEPRECATION", "1") sets an environment variable before we import tkinter. On macOS, Apple's old Tk 8.5 prints a deprecation warning unless this is set. We do this defensively even though our preferred Python uses Tk 8.6.
  • shlex safely escapes user input for inclusion in shell commands.
  • subprocess runs other programs (like git or grep) and captures their output.
  • threading lets us run a slow subprocess on a background thread so the UI does not freeze.

Step 4.2: The list of commands

COMMANDS = [
    {
        "label": "1. Commits on current branch (vs master)",
        "template": "git log --oneline master..HEAD",
        "arg_label": None,
    },
    {
        "label": "2. Files changed vs master (diff stat)",
        "template": "git diff master...HEAD --stat",
        "arg_label": None,
    },
    {
        "label": "3. Search all commit messages",
        "template": "git log --all --oneline | grep -i -E -- {arg}",
        "arg_label": "Pattern (e.g. 'BT-542|PLD'):",
    },
    {
        "label": "4. Recursive grep for pattern",
        "template": (
            "grep -rnE "
            "--exclude-dir=.git --exclude-dir=node_modules "
            "--exclude-dir=.venv --exclude-dir=venv "
            "--exclude-dir=__pycache__ --exclude-dir=.tox "
            "--exclude-dir=dist --exclude-dir=build "
            "-- {arg} ."
        ),
        "arg_label": "Pattern (e.g. 'NoEventsFoundException'):",
    },
    {
        "label": "5. Recent commits (last 20)",
        "template": "git log -n 20 --oneline",
        "arg_label": None,
    },
]

TIMEOUT_SECONDS = 60

Why a list of dictionaries? It separates data from code. To add a sixth command later, you only edit this list. None of the UI code changes. This is one of the most useful patterns in software: keep the things that change (the command list) separate from the things that do not (the UI logic).

The {arg} placeholder is a marker we will replace at runtime with the user's input. arg_label being None means "this command takes no argument."

Step 4.3: The class skeleton

class DevRunner(tk.Tk):
    def __init__(self):
        super().__init__()
        self.title("Dev Command Runner")
        self.geometry("900x600")
        self._build_ui()

Notice that DevRunner inherits from tk.Tk. That means a DevRunner instance is a Tk root window. super().__init__() calls the parent's constructor first, which sets up all the Tk machinery. Then we set a window title and a starting size, and call our own helper to build the widgets.

Step 4.4: Building the widgets

def _build_ui(self):
    top = ttk.Frame(self, padding=10)
    top.pack(fill=tk.X)

    ttk.Label(top, text="Command:").grid(row=0, column=0, sticky="w")
    self.command_var = tk.StringVar()
    self.command_box = ttk.Combobox(
        top,
        textvariable=self.command_var,
        values=[c["label"] for c in COMMANDS],
        state="readonly",
        width=60,
    )
    self.command_box.grid(row=0, column=1, columnspan=2, sticky="we", padx=5)
    self.command_box.current(0)
    self.command_box.bind("<<ComboboxSelected>>", self._on_command_change)

What is happening:

  • We create a ttk.Frame as a container for the controls and pack it at the top of the window, stretching horizontally (fill=tk.X).
  • Inside the frame we use grid to arrange labels and inputs in rows.
  • tk.StringVar() creates a string "variable" that the combobox can both read from and write to. We never actually use command_var directly here, but tkinter requires a textvariable for combobox state.
  • values=[c["label"] for c in COMMANDS] is a list comprehension. It builds the list of dropdown options by reading the label field from every command dictionary.
  • state="readonly" prevents the user from typing free text into the dropdown.
  • self.command_box.current(0) selects the first item by default.
  • bind("<<ComboboxSelected>>", callback) wires up a function to run whenever the user changes the dropdown selection. The double angle brackets denote a "virtual event." We will define _on_command_change shortly.

The rest of _build_ui follows the same pattern for the argument entry, working directory entry with a Browse button, and the Run/Clear buttons. We will not retype it all here; refer to the full source.

The output area uses scrolledtext.ScrolledText, which is a multi-line text widget with a vertical scrollbar built in:

self.output = scrolledtext.ScrolledText(
    self, wrap=tk.NONE, font=("Menlo", 11),
    background="#1e1e1e", foreground="#e6e6e6",
    insertbackground="#e6e6e6",
)
self.output.pack(fill=tk.BOTH, expand=True, padx=10, pady=(0, 10))
self.output.tag_configure("cmd", foreground="#7fd3ff")
self.output.tag_configure("err", foreground="#ff8a8a")
self.output.tag_configure("meta", foreground="#a0a0a0")
  • fill=tk.BOTH, expand=True says "stretch this to fill all remaining space, both horizontally and vertically."
  • tag_configure defines named text styles. When we insert text we can pass a tag name to apply that style. We use it to color the command line in cyan, errors in red, and metadata in gray.

Step 4.5: Reacting to dropdown changes

def _on_command_change(self, *_):
    cmd = self._selected_command()
    if cmd["arg_label"]:
        self.arg_label.configure(text=cmd["arg_label"])
        self.arg_entry.configure(state="normal")
    else:
        self.arg_label.configure(text="(no argument)")
        self.arg_var.set("")
        self.arg_entry.configure(state="disabled")

When the user picks a different command, we either show the argument hint and enable the entry, or hide the hint and disable the entry. This makes the UI self-explaining: you only see the input field when you actually need it.

The *_ in the parameter list is a Python idiom meaning "accept any extra positional arguments and ignore them." Tkinter passes an event object to the callback, but we do not need it.

Step 4.6: Running the command (the most important method)

def run_command(self):
    cmd = self._selected_command()
    template = cmd["template"]
    if "{arg}" in template:
        arg = self.arg_var.get().strip()
        if not arg:
            self._write("ERROR: this command requires an argument.\n", "err")
            return
        shell_cmd = template.replace("{arg}", shlex.quote(arg))
    else:
        shell_cmd = template

    cwd = self.cwd_var.get().strip() or os.getcwd()
    if not os.path.isdir(cwd):
        self._write("ERROR: working dir does not exist: " + cwd + "\n", "err")
        return

    self._write("$ " + shell_cmd + "\n", "cmd")
    self._write("  (cwd: " + cwd + ")\n\n", "meta")

    self.run_btn.configure(state="disabled")
    thread = threading.Thread(target=self._run_in_thread, args=(shell_cmd, cwd), daemon=True)
    thread.start()

This function does several important things:

  1. Looks up the selected command and its template.
  2. If the template needs an argument, reads the argument field, validates it is non-empty, and substitutes it into the template using shlex.quote.
  3. Validates the working directory exists.
  4. Echoes the rendered command into the output area so the user can see exactly what is about to run.
  5. Disables the Run button to prevent double-runs, and starts the actual subprocess on a background thread.
Why shlex.quote? If we just pasted the user's text into the command, an input like foo; rm -rf ~ would run rm -rf ~ as a separate command. shlex.quote wraps the input in single quotes and escapes any embedded quotes, so the shell treats the whole thing as one literal argument. This is called shell injection defense and it is a habit worth forming for any tool that builds shell strings.
Why a thread? tkinter's mainloop() runs the UI on a single thread. If we call subprocess.run directly from a button click, the UI freezes for the whole duration, and macOS shows the spinning beach ball. By running the subprocess in a separate thread, the UI stays responsive. We have to be careful, though: tkinter widgets must only be touched from the main thread. We use self.after(0, fn) to schedule UI updates back on the main thread.

Step 4.7: The background worker

def _run_in_thread(self, shell_cmd, cwd):
    try:
        result = subprocess.run(
            shell_cmd,
            shell=True,
            cwd=cwd,
            capture_output=True,
            text=True,
            timeout=TIMEOUT_SECONDS,
        )
        self.after(0, self._render_result, result)
    except subprocess.TimeoutExpired:
        self.after(0, self._render_timeout)
    except Exception as exc:
        self.after(0, self._render_error, exc)

Key things:

  • shell=True tells subprocess to run the string through a shell (so pipes like | work). Our use of shlex.quote earlier is what makes this safe.
  • capture_output=True means "collect what the program prints instead of letting it leak to the terminal." We then put it in our text widget.
  • text=True decodes the output as a string instead of raw bytes.
  • timeout=60 kills the subprocess if it takes too long.
  • self.after(0, fn, *args) schedules fn(*args) to run on the main thread "as soon as possible." This is the bridge from a worker thread back to the UI.

Step 4.8: Rendering results back to the UI

def _render_result(self, result):
    if result.stdout:
        self._write(result.stdout)
    if result.stderr:
        self._write("[stderr]\n" + result.stderr, "err")
    self._write("\n[exit code: " + str(result.returncode) + "]\n\n", "meta")
    self.run_btn.configure(state="normal")

def _write(self, text, tag=None):
    if tag:
        self.output.insert(tk.END, text, tag)
    else:
        self.output.insert(tk.END, text)
    self.output.see(tk.END)

The _write helper appends text to the output widget, optionally with a color tag, and scrolls to the bottom so the newest content is always visible.

Step 4.9: The entry point

if __name__ == "__main__":
    DevRunner().mainloop()

This single line creates a DevRunner instance and starts the tkinter event loop. mainloop() blocks until the user closes the window.

Part 5: Run and test it

Save the file as dev_runner.py in your home directory. Then in Terminal:

/Library/Frameworks/Python.framework/Versions/Current/bin/python3 ~/dev_runner.py

If you also did chmod +x ~/dev_runner.py, the shebang lets you run it directly:

~/dev_runner.py

First test: select "5. Recent commits (last 20)" and click Run. You should see 20 commit lines in the output. This requires no argument and works in any git repo, so it is the quickest sanity check.

Second test: select "4. Recursive grep for pattern", type a string you know exists in your code (a function name, a class), and click Run. You should see file paths with line numbers.

Third test: use the Browse button to switch to a different repo on your machine and re-run command 5. The same button should now show that repo's commits.

Part 6: Troubleshooting

"ModuleNotFoundError: No module named '_tkinter'"

Your Python build does not include tkinter. This is common with Homebrew's Python. Fix options:

  • Best: install Python from python.org. The installer bundles a working Tk 8.6.
  • Alternative on macOS: use /usr/bin/python3 (Apple's 3.9 build), which has Tk but is older and shows a deprecation warning.
  • Homebrew users: brew install python-tk sometimes works, but it is fiddly.

"DEPRECATION WARNING: The system version of Tk is deprecated"

This appears when you run under Apple's Python 3.9 + Tk 8.5. To suppress the warning, set TK_SILENCE_DEPRECATION=1. To actually fix the underlying issue (blank windows, ugly widgets), switch to a Python that ships with Tk 8.6.

The window appears empty

Almost always Tk 8.5 on macOS. Switch interpreters to a Tk 8.6 build (python.org installer).

Command 4 hangs or times out

You may be running it from a directory other than the repo. Or there is an unusually large directory not in the exclude list. Either change the working directory in the UI, or add another --exclude-dir to the template.

Command 1 or 2 returns nothing

You may be on a branch that has no commits beyond master, or your repo's main branch is named something else (main, not master). Edit the template strings in COMMANDS to suit your repo.

Exercises and next steps

If you want to keep going:

  1. Add a sixth command: for example "show staged changes" (git diff --cached --stat). Just append a new dictionary to COMMANDS; the UI updates automatically.
  2. Make the main branch configurable: add a "Main branch" entry near the working directory and substitute it into commands 1 and 2. Hint: use the same {arg}-style placeholder mechanism.
  3. Save output to a file: add a "Save output..." button that opens filedialog.asksaveasfilename and writes the contents of the text widget.
  4. Stream output line by line: instead of subprocess.run with capture_output=True, use subprocess.Popen and read stdout a line at a time. This is more advanced but gives a live "tail" feel for slow commands.
  5. Replace grep with ripgrep: if you have rg installed, swap the template in command 4. ripgrep already excludes .git and respects .gitignore by default.

You now have a working tkinter app and a feel for how Python wraps shell commands behind a UI. The same pattern (a list of templates plus a small UI to run them) scales surprisingly far: the same architecture works for build systems, deploy dashboards, and admin tools at real companies.
